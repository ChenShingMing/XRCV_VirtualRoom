### ⚠️ 已知風險：Quest 更新畫面可能顯示不出來

`LoadingScene` 的更新提示 UI（`UpdatePanel`）掛在 `Canvas`，Render Mode 是 **Screen Space - Overlay**。這種模式在 VR headset 裡通常不會被算進立體渲染畫面（只會出現在 PC 端的鏡射螢幕，戴上頭顯後看不到）。

也就是說 Quest 版本很可能會發生：偵測到更新、開始下載，但使用者在頭顯裡完全看不到「發現新版本」的提示畫面 / 進度條，體感上像是卡住沒反應。

**下次要做的事**：
1. 先實測確認上述推測是否成立（戴上頭顯實際看有沒有畫面）。
2. 如果確認看不到，需要規劃 VR 版的更新 UI，例如：
   - 額外做一個 world-space Canvas，掛在 XR Origin/攝影機前方固定位置
   - 或是在 `UpdateManager` 偵測到目前是 VR 平台時，切換成不同的 UI 呈現方式
3. 評估這段等待期間（下載/安裝）使用者體感上要多久，需不需要加簡單的音效或震動提示，避免使用者以為當機。

## 待辦 2：更新機制要能「選擇性」進行，避免卡住學校端 Demo

**需求**：學校端做 Demo 時，如果裝置剛好不是最新版本，不應該被強制卡住更新（Demo 現場網路可能不穩或沒有網路，強制更新會導致完全無法使用）。

**目前狀況**：`VersionManifest.IsForceUpdateRequired()` 邏輯是
```csharp
forceUpdate || CompareVersions(localVersion, minRequiredVersion) < 0
```
目前發布時 `minRequiredVersion` 跟 `latestVersion` 設成一樣（`1.0`），導致只要不是最新版就一定會被判定強制更新、畫面不會出現「略過」按鈕。

**下次要一起討論決定的方向**：
- 最簡單：往後發版時，`minRequiredVersion` 平常保持**低於** `latestVersion`，只有真的有必須升級的破壞性變更（例如更新協定不相容）才把它拉高到跟 `latestVersion` 一樣 → 純粹是發版策略調整，程式碼邏輯已經支援，不用改code。
- 是否需要加「Demo 模式 / 略過本次檢查」的開關，讓現場人員可以主動跳過整個更新檢查。
- 略過更新後要不要記住「這個版本已經略過過」，避免每次啟動都再問一次。
